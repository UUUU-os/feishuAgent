# MeetFlow OpenClaw 智能化演示增强正式任务方案

本文档是 MeetFlow 后续开发的正式任务指引之一，承接根目录
`MeetFlow_OpenClaw智能化演示增强任务方案.md` 的方案内容，并作为 `tasks.md`
中 OpenClaw/CLI/Console 演示增强工作的详细任务文档。后续实际开发、验收和文档同步
应优先以本文为准。

> 项目定位：面向飞书会议协作场景的智能工作流 Agent  
> 当前核心目标：优先增强演示效果，搭建完整、清晰、可展示的 OpenClaw 智能化流程。  
> 任务重点：会前卡片增强、会后总结卡优化、任务卡片生成、任务风险提醒卡扩充、Agent 能力增强、评测优化、后端 CLI 接入、前端现代化改造。
> 文档性质：宏观任务方案，不展开具体代码实现细节。
> 编号说明：本文内部演示增强模块统一使用 `D1-D10`，避免和项目既有 `M1-M6` 研发里程碑混淆；文中 `M3/M4/M5` 仍指 MeetFlow 已有会前、会后、任务风险提醒业务工作流。

## 1. 当前阶段的核心判断

现阶段最重要的不是继续横向扩展更多飞书 API，也不是单纯补 RAG 或做高并发，而是：

> **把现有飞书生态能力包装成一条完整、智能、可演示、可解释、可评测的 OpenClaw 工作流。**

你们项目已经具备飞书生态基础能力，包括：

- 飞书日历；
- 飞书妙记；
- 飞书群聊；
- 飞书互动卡片；
- 卡片按钮回调；
- 飞书任务；
- SDK 长连接；
- 后端 API；
- Console；
- Agent Runtime；
- ToolRegistry；
- AgentPolicy；
- 评测中心。

因此，后续重点不应是从零造功能，而应是：

1. **让已有能力更好地串起来；**
2. **让每一步输出更丰富、更像真实办公助手；**
3. **让演示链路更顺畅；**
4. **让 OpenClaw / CLI / Agent 流程更智能化；**
5. **让前端界面更现代化、更容易讲清楚项目价值。**

## 2. 新版演示主线

建议把最终演示主线设计为：

```text
OpenClaw / CLI / Console 启动项目
  ↓
读取飞书日历 / 项目历史信息
  ↓
生成智能会前卡片
  ↓
会议结束后读取飞书妙记
  ↓
Agent 分析妙记内容
  ↓
生成会后总结卡
  ↓

  ↓
生成任务卡片
  ↓
通过飞书卡片 / Console 展示任务分配
  ↓

  ↓

  ↓
```

一句话表达：

> **MeetFlow 的核心演示不是“总结会议”，而是通过 OpenClaw / CLI / Console 驱动飞书会议从会前准备、妙记复盘、任务分配、任务风险提醒到评测回放的完整智能化流程。**

## 4. D1：演示主线与 OpenClaw 流程设计

### 4.1 任务目标

把现有功能从“多个分散模块”整理成一条完整的 OpenClaw 智能化流程。

### 4.2 OpenClaw 具体定位

OpenClaw 在本项目中不能只作为口号出现，应明确定位为：

> **OpenClaw 是 MeetFlow 智能工作流的外部编排与演示入口；CLI 是 OpenClaw 调用 MeetFlow 能力的受控命令层；Console 是流程状态、证据和评测结果的可视化界面；飞书卡片是真实协作现场。**

四者关系：

```text
OpenClaw
  -> 调用 MeetFlow CLI / 工具描述
  -> 触发受控 Agent 工作流
  -> 复用 Console API / 白名单脚本 / MeetFlowAgent
  -> 经过 ToolRegistry 与 AgentPolicy
  -> 读写飞书、生成卡片、记录 trace、输出评测报告
```

### 4.3 OpenClaw 必须交付的产物

为避免“OpenClaw 只是概念”，本轮演示增强至少需要交付以下内容：

| 交付物 | 建议路径 | 说明 | 验收标准 |
|---|---|---|---|
| 统一 CLI 入口 | `scripts/meetflow_cli.py` | 封装健康检查、M3、M4、M5、评测和 demo replay | 一条命令能触发核心流程 |
| OpenClaw 工具说明 | `docs/openclaw-meetflow-tool-guide.md` | 说明 OpenClaw 如何调用 CLI、参数含义、输出格式 | 答辩时能展示调用方式 |
| OpenClaw 工具清单示例 | `config/openclaw_tools.example.json` 或文档内 JSON 示例 | 描述 `meetflow_health`、`meetflow_pre_meeting`、`meetflow_post_meeting`、`meetflow_risk_scan`、`meetflow_eval` | 工具名称、输入、输出清晰 |
| 演示命令脚本 | `docs/openclaw-demo-commands.md` | 固定主演示路径和兜底路径命令 | 团队成员能照命令复现 |
| 标准输出格式 | CLI JSON 输出 | 每次运行输出 `trace_id`、`workflow_type`、`status`、`report_path`、`safety_summary` | OpenClaw/Console 可消费 |

推荐 CLI 命令形态：

```bash
/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +health

/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +pre-meeting \
  --date today \
  --event-title "项目需求评审会" \
  --provider scripted_debug \
  --dry-run

/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +post-meeting \
  --minute "<飞书妙记链接或 token>" \
  --provider scripted_debug \
  --dry-run

/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +risk-scan \
  --backend local \
  --dry-run

/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +eval \
  --suite agent_trajectory \
  --write-report

/home/tanyd/anaconda3/envs/meetflow/bin/python scripts/meetflow_cli.py workflow +demo-replay \
  --scenario project_review \
  --dry-run
```

OpenClaw 演示时要展示的不只是“能执行命令”，而是命令执行后能看到：

- 本次 Agent 的 `trace_id`；
- 调用了哪些工具；
- 是否经过 `AgentPolicy`；
- 产出了哪张卡片或哪份报告；
- 评测分数和失败兜底路径。

### 4.4 核心任务

| 编号 | 任务 | 优先级 | 说明 | 验收标准 |
|---|---|---|---|---|
| D1-01 | 重新设计演示主线 | P0 | 从会前、会中、会后、任务、风险、评测串成闭环 | 能用一张流程图讲清 |
| D1-02 | 明确 OpenClaw 在项目中的位置 | P0 | 作为智能流程调度入口，而不是单独工具 | 有工具说明和调用示例 |
| D1-03 | 明确 CLI、Console、飞书卡片三者关系 | P0 | CLI 负责触发，Console 负责展示，飞书卡片负责真实协作 | 三个入口逻辑清楚 |
| D1-04 | 固定主演示路径 | P0 | 选择最稳定的一条流程作为正式演示 | 演示不依赖临场发挥 |
| D1-05 | 设计备用演示路径 | P0 | 如果飞书真实联调失败，可用脱敏数据和截图演示 | 有兜底方案 |
| D1-06 | 梳理每一步的智能化亮点 | P0 | 每个模块都要体现 Agent 不是简单脚本 | 答辩时能讲清楚 |
| D1-07 | 固定 CLI JSON 输出协议 | P0 | 输出 trace、状态、报告路径和安全摘要 | OpenClaw/Console 能统一消费 |

### 4.5 推荐演示叙事

演示时不要从“我们做了一个总结工具”开始，而应从：

> 企业会议协作最大的问题不是没有纪要，而是纪要无法自动转化为准备、任务、风险和后续推进。

然后展示：

1. 会前：系统主动生成背景；
2. 会后：系统自动理解妙记；
3. 任务：系统识别每个人要做什么；
4. 风险：系统持续发现推进问题；
5. 评测：系统证明自己的输出可靠；
6. OpenClaw / CLI：系统可被智能流程统一调度。

## 5. D2：会前卡片增强任务

代码级拆解、落地顺序和验收矩阵见
[D2：会前卡片智能准备增强具体改造方案](d2-pre-meeting-card-enhancement-plan.md)。

### 5.1 当前问题

会前卡片如果只是简单提醒会议时间，智能化不足。应增强为：

> **基于飞书日历、历史会议、项目文档、历史任务和轻量 RAG 的会前智能准备卡。**

### 5.2 增强目标

会前卡片应从“通知型卡片”升级为“智能准备型卡片”。

### 5.3 会前卡片建议内容

| 内容模块 | 说明 |
|---|---|
| 会议基本信息 | 会议标题、时间、参会人、会议来源 |
| 会议背景摘要 | 根据历史会议和项目资料生成背景 |
| 上次会议结论 | 回顾上次会议达成的关键决策 |
| 遗留行动项 | 列出上次会议未完成任务 |
| 历史风险 | 提醒此前出现过但未完全解决的问题 |
| 本次建议议题 | Agent 根据上下文建议本次应讨论的问题 |
| 会前 Checklist | 提醒参会人需要准备的材料 |
| Evidence Pack | 展示历史会议、任务、文档来源 |
| 快捷操作按钮 | 查看详情、开始会议、生成复盘、查看历史记录等 |

### 5.4 任务清单

| 编号 | 任务 | 优先级 | 说明 | 验收标准 |
|---|---|---|---|---|
| D2-01 | 接入历史会议信息 | P0 | 检索同项目或同主题历史会议 | 卡片能显示历史会议摘要 |
| D2-02 | 接入历史行动项 | P0 | 获取上次会议遗留任务 | 卡片能显示未完成事项 |
| D2-03 | 接入历史风险 | P0 | 获取此前出现过的风险 | 卡片能提示持续风险 |
| D2-04 | 引入轻量 RAG / Evidence Pack | P0 | 用小语料检索支持背景生成 | 卡片中有证据来源 |
| D2-05 | 生成会议背景摘要 | P0 | 让 Agent 总结本次会议背景 | 背景不是空泛描述 |
| D2-06 | 生成建议议题 | P0 | 根据历史上下文提出建议讨论项 | 能体现主动智能 |
| D2-07 | 生成会前 Checklist | P0 | 提醒材料、负责人、风险和目标 | 适合真实开会使用 |
| D2-08 | 优化会前卡片布局 | P0 | 内容分区清晰，突出重点 | 一眼看懂 |
| D2-09 | 增加证据来源区域 | P1 | 展示相关历史会议和文档 | 降低幻觉感 |
| D2-10 | 准备会前卡片演示样例 | P0 | 用于答辩和录屏 | 有截图或可复现样例 |

### 5.5 会前卡片演示重点

答辩时要强调：

> 会前卡片不是“会议提醒”，而是 Agent 基于历史会议和项目记忆主动生成的会议准备报告。

## 6. D3：会后总结卡优化任务

### 6.1 当前问题

会后总结卡如果只包含摘要和行动项，会显得普通。应增强为：

> **结构化、可追踪、带证据、带后续建议的会议复盘卡。**

### 6.2 会后总结卡建议内容

| 内容模块 | 说明 |
|---|---|
| 会议摘要 | 概括本次会议核心内容 |
| 关键结论 | 已达成一致的决策 |
| 开放问题 | 尚未解决的问题 |
| 行动项概览 | 按人、按优先级、按截止时间整理 |
| 风险提示 | 本次会议中暴露的风险 |
| 争议点 / 分歧点 | 如果妙记中存在不同观点，可提炼出来 |
| 后续建议 | Agent 给出的推进建议 |
| Evidence Pack | 关键结论对应的妙记片段或历史依据 |
| 快捷按钮 | 查看任务卡、行动项风险预检、查看完整报告 |

### 6.3 任务清单

| 编号 | 任务 | 优先级 | 说明 | 验收标准 |
|---|---|---|---|---|
| D3-01 | 优化摘要质量 | P0 | 摘要更像复盘，不是简单概括 | 摘要能说明会议价值 |
| D3-02 | 强化关键结论提取 | P0 | 明确会议达成了什么 | 结论清晰可读 |
| D3-03 | 强化开放问题提取 | P0 | 明确还没有解决什么 | 问题可继续跟进 |
| D3-04 | 按人整理行动项 | P0 | 每个人的任务清楚列出 | 能看到谁负责什么 |
| D3-05 | 增加风险提示模块 | P0 | 会后卡直接暴露风险 | 风险不需要另找 |
| D3-06 | 增加后续建议模块 | P0 | Agent 给出下一步推进建议 | 体现智能化 |
| D3-07 | 增加证据来源模块 | P1 | 关键内容关联妙记片段 | 输出更可信 |
| D3-08 | 增加完整报告入口 | P1 | 卡片内可跳转完整复盘 | 适合复杂会议 |
| D3-09 | 优化卡片视觉层级 | P0 | 摘要、任务、风险分区明显 | 卡片更易读 |
| D3-10 | 准备会后总结卡演示样例 | P0 | 用真实或脱敏妙记展示 | 可稳定演示 |

### 6.4 会后总结卡演示重点

需要突出：

> 系统不是把会议“总结成一段话”，而是把会议结构化为结论、问题、任务、风险和建议。

## 7. D4：妙记 Agent 分析与完整任务卡片生成

代码落地记录、当前实现边界和验证结果见
[D4：妙记 Agent 分析与任务卡片生成落地记录](d4-minute-agent-task-card-plan.md)。

### 7.1 任务目标

这是当前最能体现“Agent 智能化”的模块。

系统应从飞书妙记中自动识别：

- 每个人承担了什么任务；
- 任务来源于哪段讨论；
- 任务是否缺负责人；
- 任务是否缺截止时间；
- 任务优先级如何；
- 任务之间是否有关联；
- 哪些任务需要确认；
- 哪些任务可以创建为飞书任务。

### 7.2 任务卡片建议内容

| 内容模块 | 说明 |
|---|---|
| 任务标题 | 简洁描述任务 |
| 负责人 | 从妙记中识别或待确认 |
| 参与人 | 涉及协作的人 |
| 截止时间 | 从妙记中识别或标记缺失 |
| 优先级 | 高 / 中 / 低 |
| 任务来源 | 来自哪段妙记或哪个议题 |
| 任务状态 | 待确认 / 可创建 / 已创建 / 已拒绝 |
| 缺失字段 | 负责人、截止时间、描述等 |
| Agent 建议 | 是否建议创建任务、是否建议合并 |
| 操作按钮 | 确认创建、补充信息、拒绝、查看来源 |

### 7.3 任务清单

| 编号 | 任务 | 优先级 | 说明 | 验收标准 |
|---|---|---|---|---|
| D4-01 | 妙记中识别行动项 | P0 | 从会议内容中提取任务 | 行动项准确 |
| D4-02 | 按负责人聚合任务 | P0 | 让每个人看到自己的任务 | 任务分配清晰 |
| D4-03 | 识别任务截止时间 | P0 | 从文本中提取时间信息 | 可识别明确 deadline |
| D4-04 | 标记缺失字段 | P0 | 缺负责人或时间时标记 | 不盲目创建 |
| D4-05 | 生成完整任务卡片 | P0 | 每条任务以卡片形式展示 | 飞书卡片可读 |
| D4-06 | 增加任务来源说明 | P0 | 绑定妙记片段或议题 | 可追溯 |
| D4-07 | 增加任务优先级 | P1 | 根据语气、时间、风险判断 | 任务有轻重 |
| D4-08 | 增加任务合并 / 去重提示 | P1 | 与历史任务相似时提示 | 体现项目记忆 |
| D4-09 | 增加任务状态流转 | P0 | 待确认、已确认、已创建等 | 任务生命周期清楚 |
| D4-10 | 接入飞书任务创建 dry-run | P0 | 先展示计划，不一定真实写入 | 演示安全 |
| D4-11 | 接入真实任务创建 | P1 | allow-write 时创建飞书任务 | 有条件演示 |
| D4-12 | 准备任务卡片演示样例 | P0 | 展示多人任务分配 | 一眼能看出智能化 |

### 7.4 任务卡片演示重点

答辩时要强调：

> 会后 Agent 不是只总结会议，而是理解妙记中的协作意图，自动拆解出每个人的任务，并生成可确认、可追踪、可创建的任务卡片。

## 8. D5：任务风险提醒卡片扩充任务

### 8.1 当前问题

M5 任务风险提醒如果只是列出“逾期任务”，不够智能。应扩展为：

> **面向会议任务推进的风险诊断卡。**

### 8.2 任务风险提醒卡建议内容

| 内容模块 | 说明 |

### 8.3 风险类型扩充

| 风险类型 | 说明 |

| 逾期风险 | 任务超过截止时间仍未完成 |
| 临期风险 | 任务即将到期 |
| 无负责人风险 | 任务没有明确 owner |
| 无截止时间风险 | 任务没有 deadline |
| 长期未更新风险 | 任务状态长时间无变化 |

| 任务冲突风险 | 两个任务目标或负责人冲突 |
| 演示风险 | 项目启动复杂、依赖多、链路不稳定 |
| 证据不足风险 | Agent 输出缺少足够证据支撑 |

## 11. D8：后端 CLI / OpenClaw 接入任务

代码级改造方案见 [D8：后端 CLI / OpenClaw 受控接入代码改造方案](d8-cli-openclaw-integration-plan.md)。

### 11.1 任务目标

后端 CLI 不只是启动脚本，而应成为项目亮点之一：

> **把现有 Agent、Console facade 和白名单脚本能力包装成 OpenClaw / CLI 可调用的受控智能工作流入口。**

这里要特别避免一个风险：CLI 不能因为“为了演示方便”而绕过当前项目已经建立的
`ToolRegistry`、`AgentPolicy`、幂等和 `allow_write` 安全边界。

### 11.2 CLI 应支持的能力

| CLI 能力 | 说明 |
|---|---|
| 环境检查 | 检查飞书配置、模型配置、Console、SDK 状态 |
| 会前卡片生成 | 触发 M3 会前准备 |
| 妙记复盘 | 输入妙记链接，触发 M4 会后总结 |
| 任务卡片生成 | 根据妙记生成任务卡 |
| 任务风险提醒 | 触发 M5 任务风险扫描 |
| 评测运行 | 运行 Agent eval suite |
| 演示回放 | 一键运行 demo 流程 |
| OpenClaw 调度 | 作为外部智能流程的工具入口 |

### 11.3 CLI 安全边界

CLI / OpenClaw 接入必须遵守以下边界：

```text
OpenClaw / CLI
  -> scripts/meetflow_cli.py
  -> Console API facade / 现有白名单脚本 / MeetFlowAgent.run()
  -> WorkflowRouter
  -> WorkflowContextBuilder
  -> MeetFlowAgentLoop
  -> ToolRegistry
  -> AgentPolicy
  -> FeishuClient / Storage
```

明确禁止：

- CLI 直接调用 `FeishuClient.send_*`、`FeishuClient.create_*` 等真实写接口；
- CLI 直接写业务 SQLite 表来伪造任务、风险或回调结果；
- CLI 接收任意 shell 命令或任意 Python 表达式；
- CLI 默认执行真实写操作；
- CLI 在日志、stdout、报告中输出 token、secret、refresh token、API key。

必须保留：

- 默认 `dry-run`；
- 真实写操作必须显式 `--allow-write`；
- 写操作必须有 `idempotency_key`；
- 任务创建、消息发送、卡片发送必须经过 `AgentPolicy`；
- 所有命令输出必须包含 `trace_id`，方便 Console 和评测报告串联；
- 失败要返回真实原因，不能为了演示伪造成成功。

CLI 输出建议统一为 JSON：

```json
{
  "status": "success",
  "workflow_type": "post_meeting_followup",
  "trace_id": "trace_xxx",
  "dry_run": true,
  "allow_write": false,
  "report_path": "storage/reports/...",
  "agent_trace_path": "storage/reports/evaluation/...",
  "safety_summary": {
    "policy_checked": true,
    "write_blocked_or_confirmed": true,
    "idempotency_key_present": true
  }
}
```

```text
2. 触发会前准备
   Agent 读取历史会议和项目记忆，生成会前卡片

3. 展示会前卡片
   包括会议背景、历史结论、遗留任务、建议议题、Checklist、Evidence

4. 输入飞书妙记链接
   Agent 读取妙记并分析会议内容

5. 展示会后总结卡
   包括摘要、关键结论、开放问题、行动项、风险点、后续建议

6. 展示任务卡片
   按人分组展示每个人的任务、截止时间、来源、状态和操作

7. 展示 Agent Trace
   展示 Agent 检索了什么、调用了什么工具、为什么生成这些任务

8. 触发 M5 任务风险提醒
   Agent 根据任务状态、历史会议和项目记忆生成任务风险提醒结果

9. 展示任务风险提醒卡
   包括高/中/低风险、来源、原因、影响范围、建议动作

10. 展示评测中心
    展示会后总结质量、任务识别、风险识别、Evidence Coverage、安全策略和工具轨迹

11. 展示 CLI / OpenClaw
    用命令行触发同样流程，说明项目可以被 OpenClaw 调度

12. 总结项目价值
    飞书生态闭环 + Agent 智能化 + 证据支撑 + 任务风险提醒 + 可评测
```

## 17. 最终答辩表达

可以这样表达：

> MeetFlow 不是一个简单的会议总结机器人，而是一个完整接入飞书生态的会议工作流 Agent。它通过 OpenClaw / CLI / Console 统一调度会议流程：会前阶段，Agent 结合飞书日历、历史会议和项目记忆生成会前准备卡；会后阶段，Agent 读取飞书妙记，提炼会议摘要、关键结论、开放问题、行动项和风险点；随后系统按负责人生成任务卡片，明确每个人要做什么、什么时候做、依据是什么；最后 M5 任务风险提醒 Agent 会结合任务状态、历史会议和 Evidence Pack，生成任务风险提醒卡，帮助团队发现逾期、临期、无人负责、反复出现等问题。

> 项目的智能化不只体现在大模型生成文本，而体现在完整的工作流编排：ContextBuilder 聚合飞书和历史上下文，ToolRegistry 管理飞书工具和检索工具，AgentPolicy 控制写入安全和幂等，Evidence Pack 降低幻觉，Eval Center 评估总结质量、任务识别、风险识别、工具调用和安全策略。前端 Console 则把这一整套流程可视化，让评委能看到 Agent 从会前准备到会后任务推进的完整闭环。

一句话压缩版：

> **MeetFlow 的核心创新是用 OpenClaw / CLI / Console 驱动飞书会议从会前准备、妙记复盘、任务卡片、任务风险提醒到评测回放的完整智能化办公流程。**

## 18. 最终结论

当前最重要的任务不是继续增加零散功能，而是围绕演示效果做系统性增强：

1. 会前卡片要更智能，体现历史会议和轻量 RAG；
2. 会后总结卡要更丰富，体现结构化复盘；
3. 任务卡片要更完整，体现 Agent 对妙记的任务理解能力；
4. 任务风险提醒卡要更清晰，体现持续跟踪和管理价值；
5. Agent 功能要更可解释，体现工具调用、策略和 Trace；
6. 评测要更完整，证明系统不是纯 Demo；
7. CLI / OpenClaw 要接入受控 Agent / Console / 白名单脚本入口，成为项目亮点；
8. 前端要现代化，承担主控台和演示中心的作用。

最终要让评委看到的不是：

> “这是一个会议总结工具。”

而是：

> **“这是一个完整接入飞书生态，并通过 OpenClaw 智能调度的会议协作 Agent。”**
