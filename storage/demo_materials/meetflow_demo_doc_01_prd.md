# MeetFlow 比赛演示资料 01：产品目标与验收指标

> 这份文档用于 MeetFlow RAG 演示。核心结论：MeetFlow 要把会议前准备、会议后任务沉淀、任务风险跟进，从人工跨应用查找，变成飞书内自动触发、卡片确认、可追溯闭环。

## 1. 演示目标

MeetFlow 的比赛演示目标不是展示单个 API 能调用成功，而是展示一个可以在真实飞书环境后台运行的垂直 Agent 系统。

演示时只需要用户在飞书中操作：

- 创建或修改会议日程。
- 编辑项目相关云文档。
- 开会并生成妙记。
- 在群卡片中确认或拒绝待办。

系统应自动完成：

- M3 会前背景卡：在会议开始前，基于日程、RAG 文档、历史妙记、任务状态生成背景卡。
- RAG 文档更新：监听已订阅文档变更，将最新内容刷新到本地知识索引。
- M4 会后总结卡：会议结束并产生妙记后，抽取决策、待办、开放问题，推送会后卡片。
- M5 风险巡检：定期扫描任务，发现逾期或长时间未更新的事项，并回链到会议来源。

## 2. 用户痛点

| 痛点 | 人工流程 | MeetFlow 改进 |
| --- | --- | --- |
| 会前不知道该看什么 | 打开日历、搜索文档、翻历史妙记、检查任务 | M3 自动生成待读资料、当前问题、风险提示 |
| 会后待办容易丢 | 人工读妙记、复制待办、补负责人和截止时间 | M4 抽取待办并在卡片中人工确认 |
| 文档变化不同步 | 人工记得更新会议材料 | RAG 监听文档变化并刷新索引 |
| 任务风险发现晚 | 人工定期查任务列表 | M5 后台扫描并推送风险卡 |

## 3. 验收指标

比赛录屏时建议展示以下指标：

| 指标 | 目标值 | 说明 |
| --- | ---: | --- |
| 会前资料整理时间 | 从约 18 分钟降到约 3 分钟 | M3 自动聚合背景 |
| 会后待办整理时间 | 从约 22 分钟降到约 4 分钟 | M4 自动抽取并保留确认 |
| RAG 更新延迟 | 文档修改后进入索引刷新任务 | 通过长连接事件或兜底扫描触发 |
| 错误任务创建 | 低置信度任务不直接创建 | 缺负责人/截止时间时进入人工确认 |
| 风险追溯 | 风险卡能回链到妙记来源 | M5 关联 M4 task mapping |

## 4. 本次录屏重点

本次录屏建议按以下顺序展示：

1. 启动 `scripts/meetflow_up.py`，展示后台一键运行。
2. 修改这份文档中的某句话，展示 RAG 文档更新。
3. 创建一个 10 分钟后的会议，展示 M3 会前背景卡。
4. 开一个带录制的会议，按会议脚本讨论，生成真实妙记。
5. 日程结束后展示 M4 会后总结卡，点击按钮创建或拒绝任务。
6. 运行或等待 M5 风险扫描，展示任务风险卡和来源证据。

## 5. 风险与边界

- 飞书妙记是否生成取决于真实会议录制和飞书后台处理，不建议用少于 5 分钟的会议测试。
- M4 当前优先基于日程结束后扫描妙记，不是直接监听“真实会议离会”事件。
- 如果妙记没有 AI 总结或待办，M4 仍可基于逐字稿/文本进行本地抽取，但效果取决于文本中是否有明确待办表达。
- 所有真实写操作都应在测试群中进行，并显式开启 `--allow-write`。
