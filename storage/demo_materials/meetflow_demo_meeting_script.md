# MeetFlow 比赛演示会议脚本

> 用法：你和同学在飞书会议中照着这份脚本自然读出来，会议保持 8 到 12 分钟，并开启录制和妙记。说话时尽量读出“负责人”和“截止时间”，这样 M4 更容易抽取待办。

## 会议设置

- 会议名称：MeetFlow 比赛演示评审会
- 建议时长：10 到 15 分钟
- 参会人：你 + 同学
- 必须操作：开启录制，确认会后能生成妙记

## 对话脚本

张三：我们今天评审 MeetFlow 的比赛演示闭环，重点不是单个接口，而是看 M3 会前背景、RAG 文档更新、M4 会后总结、M5 风险巡检能不能串起来。

李四：我先确认一下会前背景。M3 应该在会议前自动读到产品目标、架构说明、测试场景和历史会议纪要，然后在测试群里发一张背景卡。

张三：对。我们希望背景卡里至少有三类内容：第一是待读资料，第二是当前开放问题，第三是演示风险。比如妙记生成延迟、日程没结束导致 M4 不触发、按钮创建任务失败，这些都应该被提醒。

李四：关于 RAG，我刚刚修改了测试场景文档，新增了 M4 触发策略说明。系统应该能通过文档变更事件或兜底扫描，把新内容更新到本地知识索引里。

张三：这个作为第一个结论：RAG 文档更新是比赛演示的基础能力。决定保留文档监听和 index_jobs 队列，方便录屏时证明系统能捕捉变化。

李四：第二个结论是 M4 会后总结不应该直接创建任务，而是先发卡片让人确认。因为妙记里可能有模糊待办，如果没有负责人或截止时间，直接创建任务会有风险。

张三：同意。明确待办我们现在说三条。第一条，张三负责整理 M4 效果测试报告，截止明天 18 点前完成。

李四：第二条，李四负责补充 RAG 文档更新说明，截止本周五前完成。

张三：第三条，张三负责验证 M4 卡片按钮能真实创建和拒绝任务，截止今天下班前完成。

李四：还有一个风险点。M5 风险巡检如果直接提醒所有人，可能会打扰真实用户。所以我们决定比赛阶段只把风险卡发到测试群。

张三：这个作为第三个结论：M5 风险卡先发测试群，确认稳定后再扩展到项目群。

李四：开放问题有两个。第一，系统是否要支持没有日程的临时会议妙记？如果没有绑定日程，现在可能不会自动触发 M4。

张三：第二，M4 是等日程结束再查妙记，还是监听妙记生成后立刻触发？当前为了稳定演示，先采用日程结束后立即扫描妙记。

李四：总结一下，今天的决策是：M3 用 RAG 自动准备会前背景，M4 用妙记生成会后卡片并让人确认任务，M5 扫描任务风险并回链来源妙记，所有写操作都只在测试群和人工确认后执行。

张三：好的。会后我们看 M4 卡片是否能抽出刚才三条待办，并测试创建任务和拒绝任务按钮。
