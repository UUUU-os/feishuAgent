# MeetFlow 比赛演示资料 03：测试场景、风险与验收脚本

> 这份文档用于 RAG 和录屏验收。核心结论：演示不能只证明接口能跑通，必须证明系统在正常、低上下文、缺字段、无待办、风险逾期等场景下都有合理行为。

## 1. 关键测试场景

| 编号 | 场景 | 预期行为 |
| --- | --- | --- |
| T1 | 会前资料充分 | M3 生成背景卡，包含待读资料、问题、风险 |
| T2 | 会前资料不足 | M3 不伪造资料，提示上下文不足 |
| T3 | 文档发生修改 | RAG 索引刷新，新内容可被检索 |
| T4 | 妙记有明确待办 | M4 抽取待办并生成待确认卡 |
| T5 | 妙记只有决策 | M4 不误建任务 |
| T6 | 待办缺负责人/截止时间 | M4 进入人工补字段 |
| T7 | 任务逾期 | M5 推送风险卡并回链来源妙记 |

## 2. 建议录屏数据

本次录屏建议创建一个会议：

- 会议名称：`MeetFlow 比赛演示评审会`
- 会议描述：`评审 M3 会前背景、M4 会后总结、M5 风险巡检和 RAG 文档更新闭环。`
- 会议时长：10 到 15 分钟。
- 参会人：你和一位同学。
- 必须开启会议录制和妙记。

建议会议中明确说出以下待办：

- 张三负责整理 M4 效果测试报告，截止明天 18 点前。
- 李四负责补充 RAG 文档更新说明，截止本周五前。
- 风险扫描策略暂不直接提醒全员，先只发到测试群。

建议会议中明确说出以下开放问题：

- 是否需要支持非日程会议的妙记触发？
- M4 是否应该在妙记生成后立刻触发，还是等待日程结束？

## 3. RAG 文档修改演示

录屏时可以在这份文档中追加一句：

```text
录屏测试：RAG 已捕捉到文档变化，并将 M4 触发策略更新为日程结束后立即扫描妙记。
```

然后观察后台日志中的 RAG 刷新任务，或查询：

```bash
sqlite3 storage/knowledge/knowledge.sqlite \
  "SELECT job_id,resource_id,resource_type,reason,status,last_error,datetime(updated_at,'unixepoch','localtime') FROM index_jobs ORDER BY updated_at DESC LIMIT 10;"
```

## 4. 会后 M4 卡片验收

M4 卡片应展示：

- 会议总结。
- 决策列表。
- 待办列表。
- 每条待办的负责人和截止时间。
- 创建任务 / 拒绝任务按钮。

点击创建任务后，应能在飞书任务中看到真实任务。

点击拒绝后，应记录拒绝状态，避免重复创建。

## 5. 风险说明

- 如果会议太短，飞书可能只生成文字记录，不生成 AI 总结。
- 如果会议没有开启录制，`vc +recording` 可能找不到 `minute_token`。
- 如果日程还没结束，当前 M4 自动扫描可能不会立刻触发。
- 如果测试群没有机器人，卡片不会送达。
- 如果 user 授权缺少 scope，日历、云文档、任务相关接口可能失败。
