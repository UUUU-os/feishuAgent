# 飞书 Agent 需求分析文档

## 1. 项目背景

### 1.1 赛事背景

飞书校园 AI 挑战赛课题一聚焦“办公场景驱动的智能知识助手”，强调的不只是“能回答问题”，而是要基于飞书文档、会议妙记、消息、任务等真实办公载体，做出具备主动触发、主动分发、可验证价值的知识服务产品。

在企业协作场景中，真正高价值的信息通常呈现出三个特点：

- 信息分散：会议纪要在妙记，决策背景在文档，执行动作在任务，临时共识在群聊。
- 信息时效性强：用户往往不是“不知道答案”，而是不知道“现在最该看哪一份信息”。
- 闭环成本高：会前找资料、会中对齐历史、会后拆解待办，常常依赖人工完成，容易遗漏。

因此，单纯的搜索或通用问答工具并不能很好地解决问题。员工往往还要自己判断：

- 该搜哪类对象，是文档、会议还是消息？
- 哪些是最新结论，哪些已经过期？
- 会后 Action Item 是否已经进入任务系统并分配负责人？

本项目需要构建的，不是“另一个聊天机器人”，而是一个面向办公流的主动型知识协作 Agent。

### 1.2 方向选择结论

综合可行性、创新性、演示效果和落地价值，建议本项目优先选择：

**方向 B：会议与项目的全链路伴侣**

并吸收方向 D“待办闭环”的能力，形成一个更完整的产品方向：

**项目名称建议：MeetFlow - 飞书会议知识闭环 Agent**

### 1.3 选择该方向的原因

#### 可行性高

- 会议是飞书生态内天然高频场景，容易与日历、妙记、文档、群消息、任务联动。
- “会前 - 会中 - 会后”链路天然具备主动触发点，便于实现赛事要求中的定时触发、事件触发和阈值触发。
- Demo 可控，容易通过 1-2 条核心链路在比赛周期内做出可运行版本。

#### 创新性足够

- 相比普通问答 Bot，本项目强调“背景知识主动推送 + 会后行动自动落地 + 风险持续追踪”。
- 相比简单纪要总结，本项目进一步把会议知识转化为任务、负责人、截止时间、关联依据，形成执行闭环。
- 知识产物不只是回答文本，而是“会前卡片、会后任务卡片、风险提醒卡片、CLI 提示”这类场景化产物。

#### 实用价值强

- 直接命中团队协作中的核心痛点：信息对齐慢、行动项遗漏、上下文缺失、责任不清。
- 适用于研发周会、项目例会、需求评审、跨部门同步会等常见办公场景。
- 易于设计效果指标，如会后任务落地率、补充上下文点击率、Action Items 漏检率等。

### 1.4 对 nanobot 设计的参考思路

本项目可以借鉴 nanobot 的轻量 Agent 设计理念，但不照搬其产品形态，而是抽取其适合比赛落地的架构思想：

- 通道解耦：将飞书消息、文档、妙记、任务作为不同输入输出通道，由统一 Agent 编排层处理。
- 记忆分层：短期会话上下文、项目长期记忆、结构化历史摘要分层存储，避免所有信息直接塞进对话。
- 调度驱动：借鉴 nanobot 的 cron/schedule 思想，让 Agent 支持定时巡检、事件回调和延迟跟进。
- 工具化执行：将“检索文档、读取妙记、生成卡片、创建任务、追踪状态”都封装为可组合工具。
- 可审计：所有关键结论都保留来源链接与证据片段，降低幻觉风险。

## 2. 项目目标

### 2.1 总体目标

构建一个基于 OpenClaw/CLI 与飞书生态的主动式会议知识协作 Agent，使其能够围绕会议场景完成“会前背景知识注入、会后任务自动落地、会后进展持续跟踪”三类核心能力，提升团队的信息流转效率与执行闭环质量。

### 2.2 业务目标

- 降低会前人工检索历史信息的成本。
- 降低会后人工整理纪要和拆解任务的成本。
- 提高 Action Item 的分派准确率与跟进及时性。
- 让知识服务从“被动问答”升级为“主动触达”。

### 2.3 产品目标

- 至少具备一种稳定的主动触发能力，优先实现“事件触发 + 定时触发”双触发模式。
- 输出的知识产物必须可溯源，附带文档/妙记/消息依据。
- 形成可演示的闭环体验：会前推送 -> 会后生成任务 -> 后续风险提醒。

### 2.4 用户目标

目标用户主要包括：

- 项目经理：需要快速掌握背景、推动事项闭环。
- 研发/产品负责人：需要在会议前后快速对齐历史决策与当前执行项。
- 团队成员：需要明确“我需要做什么、为什么做、截止时间是什么、参考材料在哪里”。

## 3. 核心功能分解

### 3.1 功能总览

本项目建议拆分为五个一级功能模块：

1. 会前背景知识组装
2. 会后纪要理解与 Action Item 抽取
3. 任务自动落地与知识分发
4. 进展追踪与风险预警
5. 记忆沉淀与效果评估

### 3.2 模块一：会前背景知识组装

#### 功能目标

在会议开始前，自动汇总该会议最相关的历史知识，减少参会者手动翻找背景资料的成本。

#### 输入

- 日历事件/会议标题
- 参会人信息
- 历史相关文档
- 上次会议妙记
- 项目相关任务与评论

#### 输出

- 会前背景知识卡片
- 会议摘要包（简版）
- 关键决策与未完成事项列表

#### 子功能

- 会议主题识别：基于会议标题、描述、参与人识别会议所属项目或议题。
- 关联资料召回：从飞书文档、妙记、群聊记录、历史任务中召回相关信息。
- 高密度知识提炼：输出“上次结论 / 当前问题 / 待确认事项 / 关键风险 / 参考资料链接”。
- 主动推送：在会议开始前 15-30 分钟推送到群或私聊。

#### 设计重点

- 不追求“信息全”，而追求“参会前必须知道的最小知识集”。
- 输出应强调证据链，每一条结论都附来源。
- 当日程标题或描述不足时，系统应结合参会人、附件、历史相似会议、项目记忆和近期共享文档做 query 增强；置信度不足时只输出“可能相关资料”或请求用户确认，不编造会议背景。
- 会前资料需要支持飞书文档、表格、妙记、任务和消息等不同形态。首版以轻量 RAG 为主，保留原始来源、更新时间、文档位置和结构化元数据，再做语义/关键词混合检索。
- RAG 检索应以受控工具形式接入 Agent。模型需要证据时调用检索工具，工具只返回压缩后的证据包和来源引用；完整文档内容不默认拼入模型上下文，避免长文档挤占窗口并降低幻觉风险。
- 首版不建设全量企业知识库，而是聚焦会议相关知识域：日程附件、历史会议纪要、指定知识库空间、项目群近期分享资料和已被 MeetFlow 引用过的文档。
- 文档和表格会持续变化，索引需要支持增量更新。M3 阶段先通过更新时间和内容 checksum 判断是否重建；后续再接入飞书事件订阅做实时更新。

### 3.3 模块二：会后纪要理解与 Action Item 抽取

#### 功能目标

在会议结束后，自动处理妙记/纪要，识别结论、待办、负责人、截止时间和阻塞项。

#### 输入

- 飞书妙记内容
- 文档纪要
- 会中补充消息

#### 输出

- 会后总结卡片
- Action Item 结构化列表
- 决策记录与待澄清问题
- 相关背景资料链接和待确认任务提示卡片

#### 子功能

- 纪要清洗：去除口语赘述、重复表述和与任务无关的闲聊信息。
- 任务抽取：抽取“事项 - 负责人 - 截止时间 - 优先级 - 依赖背景”。
- 决策抽取：提炼最终共识、方案取舍和开放问题。
- 待确认卡片：所有任务创建前都进入人工审核，集中展示负责人、截止时间等关键字段，支持确认创建、修改信息和拒绝创建；公网回调模式下由飞书卡片回调服务处理，本地演示模式下优先尝试 WebSocket 接收回复确认，失败时回退到群消息轮询，确认创建仍需通过任务写入策略校验。
- 证据回链：为每个 Action Item 附上对应妙记片段或文档块链接。

#### 设计重点

- 对“负责人缺失”“时间缺失”的待办要单独标记，不强行编造。
- 对模糊表述使用待确认状态，而不是直接写入正式任务。

### 3.4 模块三：任务自动落地与知识分发

#### 功能目标

将会议中形成的行动项直接同步到任务系统或统一推进表，并把执行所需背景一并分发给责任人。

#### 输入

- Action Item 结构化结果
- 任务系统模板
- 关联知识链接

#### 输出

- 飞书任务
- 责任人通知卡片
- 团队推进表更新记录

#### 子功能

- 任务生成：基于抽取结果创建飞书任务或写入统一表格。
- 字段补齐：根据上下文推断优先级、所属项目、参考资料。
- 通知分发：向责任人私聊或群内发送卡片，展示“要做什么 + 为什么做 + 看哪份材料”。
- 人工确认机制：高风险任务进入“待确认”队列，由用户一键确认后再写入正式系统。

#### 设计重点

- 自动化落地必须可控，M4 任务创建默认只支持“确认后创建”。
- 推送内容不只给任务标题，必须附背景知识链接，否则执行人仍需二次检索。

### 3.5 模块四：进展追踪与风险预警

#### 功能目标

会后持续跟踪任务状态，识别逾期、无人认领、长期阻塞等风险，并在合适时机主动提醒。

#### 输入

- 飞书任务状态
- 任务评论/文档评论
- 后续会议纪要
- 手工更新记录

#### 输出

- 风险提醒卡片
- 周期性推进简报
- 重点事项对账表

#### 子功能

- 进展回收：定时获取任务状态与评论变化。
- 风险识别：识别逾期、状态长期不更新、依赖未完成、多人责任不清等问题。
- 阈值提醒：如超过 3 天未更新、距离截止时间 24 小时仍未完成则主动提醒。
- 自动对账：将会议待办、任务系统状态、项目文档状态进行交叉校验。

#### 设计重点

- 风险提醒要“少而准”，避免变成噪音通知。
- 优先向负责人私聊提醒，对管理者输出聚合视图。

### 3.6 模块五：记忆沉淀与效果评估

#### 功能目标

将零散会议知识沉淀为可复用的项目记忆，并对 Agent 产出的准确性与效率提升进行量化评估。

#### 输入

- 历史会议摘要
- 历史任务执行情况
- 用户反馈

#### 输出

- 项目长期记忆库
- 评测样本集
- 效果验证报告指标数据

#### 子功能

- 项目记忆沉淀：沉淀项目背景、角色、关键决策、共识口径。
- 评测集建设：人工标注若干会议样本，对比 Agent 抽取结果。
- 指标统计：统计准确率、任务认领率、卡片点击率、时间节省值等。
- 反馈学习：收集“推送是否有用”“任务是否准确”的用户反馈。

#### 设计重点

- 记忆应分层：短期会话记忆、项目长期记忆、结构化历史记录分开存储。
- 评估要覆盖“准确性 + 接受度 + 效率”三维，而不是只看摘要好不好看。

## 4. 业务流程

### 4.1 总体业务流程

本项目整体围绕三个阶段运行：

1. 会前准备
2. 会后处理
3. 会后跟踪

### 4.2 会前流程

#### 流程说明

当系统检测到即将开始的会议时，自动启动会前知识准备流程。

#### 业务步骤

1. 读取日历事件，识别会议标题、时间、参与人、所属群组。
2. 基于会议主题召回相关文档、上次会议妙记、历史任务和评论。
3. 对召回内容做去重与可信度排序。
4. 生成“会前背景知识卡片”。
5. 在会议开始前推送到会议群或关键参会人私聊。

#### 主动触发方式

- 定时触发：会议开始前 15-30 分钟自动执行。
- 事件触发：日历事件创建或更新时重新生成背景卡片。

### 4.3 会后流程

#### 流程说明

会议结束且妙记生成后，系统自动处理纪要内容，提炼结论与行动项，并完成任务分发。

#### 业务步骤

1. 监听妙记就绪事件或轮询妙记状态。
2. 拉取妙记内容与相关纪要文档。
3. 抽取会议结论、待办事项、负责人、截止时间和阻塞点。
4. 对高风险或低置信度事项标记“待确认”。
5. 经人工确认后创建飞书任务或写入推进总表。
6. 向责任人发送包含背景链接的任务卡片。
7. 向群内推送会后总结卡片。

#### 主动触发方式

- 事件触发：妙记转写完成后自动执行。
- 人工触发兜底：用户在群中发送指令手动生成会后卡片。

### 4.4 会后跟踪流程

#### 流程说明

对于已创建的任务，系统会周期性巡检并进行状态对账和风险提醒。

#### 业务步骤

1. 定时拉取任务状态、截止时间、评论与更新记录。
2. 与会议抽取出的原始 Action Item 进行对账。
3. 识别未认领、逾期、长期无更新、阻塞未解除等情况。
4. 生成风险提醒卡片并推送给责任人或负责人。
5. 汇总形成周期性推进简报，发送给项目群或管理者。

#### 主动触发方式

- 定时触发：每天固定时间巡检一次。
- 阈值触发：超过设定未更新时长或临近截止时自动提醒。

### 4.5 核心异常流程

#### 场景一：妙记未就绪

- 系统识别妙记状态未完成。
- 自动进入重试队列，延后若干分钟再次拉取。
- 若超过阈值仍未生成，则仅基于会中消息和文档先输出“临时会后摘要”。

#### 场景二：Action Item 信息不完整

- 若缺少负责人或截止时间，不直接创建正式任务。
- 输出待确认卡片，请主持人或 PM 一键补充后写入任务系统。

#### 场景三：召回资料过多

- 优先保留最近一次会议、当前迭代文档、未关闭任务三类信息。
- 对其余材料只保留扩展阅读链接，不在主卡片中展开。

## 5. 推荐的 Agent 架构形态

### 5.1 架构原则

- 轻量优先：优先完成可跑通的单 Agent + 工具编排，不做过重的多 Agent 炫技设计。
- 可解释优先：所有生成结果附带来源。
- 主动服务优先：至少覆盖一种事件触发与一种定时触发。
- 人机协同优先：关键动作支持确认与回退。
- 骨架优先：每个核心工作流保留确定性阶段和校验规则，LLM 只在受控工具编排槽位中补证据、抽取和判断，避免全流程自由发挥。

### 5.2 架构分层

#### 1. 接入层

负责接入飞书生态能力：

- 飞书消息/群聊
- 日历事件
- 飞书文档
- 飞书妙记
- 飞书任务

#### 2. 编排层

由 OpenClaw Agent 或 CLI Agent 负责统一调度：

- 接收触发事件
- 选择流程模板
- 调用检索、总结、抽取、创建任务等工具

#### 3. 知识处理层

负责知识召回、证据排序与结构化抽取：

- 文档/妙记召回
- 关键信息摘要
- Action Item 抽取
- 风险识别

#### 4. 执行层

负责将结论写回飞书：

- 发送卡片
- 创建任务
- 更新推进表
- 发送提醒

#### 5. 记忆与评估层

负责项目长期记忆和效果统计：

- 会议历史摘要
- 项目长期记忆
- 指标埋点与评估报表

### 5.3 为什么不建议一开始做复杂多 Agent

虽然多 Agent 设计有展示感，但在比赛周期内，真正影响落地效果的关键通常不是 Agent 数量，而是：

- 数据接入是否稳定
- 触发链路是否可靠
- 输出是否可溯源
- 确认后创建任务是否可控

因此建议首版采用：

**一个主编排 Agent + 多工具插件 + 定时/事件调度器**

后续如需增强，可再拆分成：

- 检索 Agent
- 摘要 Agent
- 任务落地 Agent
- 风险巡检 Agent

## 6. 首版 Demo 建议范围

### 6.1 必做能力

- 会前背景知识卡片生成
- 会后 Action Item 抽取
- 飞书任务确认后创建
- 每日风险巡检提醒

### 6.2 可选增强

- CLI 本地提醒模式
- 统一推进总表
- 用户反馈按钮
- 低置信度待确认工作流

### 6.3 Demo 叙事建议

Demo 可围绕一个真实项目例会来展示：

1. 会议前，Agent 主动推送背景卡片。
2. 会议后，Agent 自动产出总结和任务。
3. 两天后，Agent 发现一项任务无人更新，主动提醒。

这样能完整体现“知识获取 - 知识应用 - 效果证明”的比赛要求。

## 7. 文档结论

本项目建议选择“会议与项目的全链路伴侣”作为主方向，并吸收“待办中枢与自动对账”的闭环能力，形成一个兼顾创新性、实用性与可落地性的主动知识服务 Agent。

其核心价值不在于“回答得多聪明”，而在于：

- 在会前，把人最需要知道的背景主动送到眼前；
- 在会后，把讨论结果转化成可执行任务；
- 在执行中，持续对账并提醒风险；
- 在长期，沉淀成团队可复用的项目记忆。

这一路径最符合飞书办公场景的产品价值，也最适合在比赛中做出“可运行、可验证、可讲清楚”的成果。
